## 一、MVP 概述

### 1.1 核心使命

> 证明自动化对练+裁判能可靠地发现 Agent 的真实问题，并且配置和使用的门槛是可接受的。

### 1.2 验证假设

| 假设 | 验证标准 |
|------|---------|
| H1 价值假设：评测能发现真实问题 | 裁判 Checklist 判定与人工抽检一致率达到可接受水平 |
| H2 体验假设：配置和使用门槛可接受 | 用户能在合理时间内完成首次配置并跑通一轮批测 |

### 1.3 关键设计决策摘要

| 项 | 决策 |
|----|------|
| 项目与 Agent 关系 | 一个项目支持多个 Agent 版本（同一 Agent 不同配置/版本的 A/B 对比） |
| 用例类型 | 仅角色驱动（L1） |
| 裁判评判时机 | 仅终局评判（对话结束后统一评判） |
| 裁判评判体系 | 双轨：Checklist + Evaluation（简化版） |
| 通过条件 | 固定规则：所有🔴必过 Checklist 通过 + Evaluation 均分 ≥ 阈值 |
| 模型配置层级 | 项目级，对练模型与裁判模型分别配置 |
| 对话终止机制 | 三层：Agent 挂断信号 / 对练机器人自然结束 / 最大轮次兜底 |
| API 协议 | 仅基础协议（纯对话接口） |

## 二、信息架构

### 2.1 整体结构

```
项目列表页
  └─ 项目工作台（顶部 Tab 切换）
       ├─ Tab 1：Agent 版本管理     ← 定义"测谁"
       ├─ Tab 2：实验设计            ← 定义"怎么测"
       │    ├─ 用例集（角色驱动）
       │    └─ 裁判配置（Checklist + Evaluation）
       ├─ Tab 3：模型配置            ← 对练 / 裁判模型分别配置
       └─ Tab 4：批测中心            ← "去测" + "看结果"
            ├─ 批测列表
            └─ 批测详情页（独立页面，从列表点击进入）
```

### 2.2 页面清单

| # | 页面 | 层级 | 说明 |
|---|------|------|------|

| P2 | 项目工作台 | 二级 | 项目内所有工作区的容器 |
| P2-T1 | Agent 版本管理 | Tab | 被测 Agent 的多版本配置 |

```
│  │ }                                                      │    │

│  （当该路径的值等于此值时，判定 Agent 主动结束对话）           │
│  ☐ Agent 无挂断信号（仅依赖对练机器人结束或最大轮次兜底）     │
│                                                                │

└──────────────────────────────────────────────────────────────┘
```

### 5.4 连接测试交互

点击 [连接测试] 后：

1. 系统用配置的模板构造一条请求，`{{session_id}}` 填入随机 UUID，`{{user_message}}` 填入固定测试语句（如"你好"）
2. 发送请求，展示：
   - 请求状态（成功 / 失败 + 错误信息）
   - 原始响应 JSON（折叠展示）
   - 按 JSONPath 解析出的 Agent 回复内容（高亮展示）
   - 挂断信号解析结果（如配置了的话）
3. 用户确认解析结果正确后保存

### 5.5 字段规则

| 字段 | 必填 | 校验 |
|------|------|------|
| 版本名称 | 是 | 项目内唯一 |
| Endpoint | 是 | 合法 URL 格式 |
| Token | 否 | 无（可为空，部分 API 无需认证） |
| 请求体模板 | 是 | 必须包含 `{{session_id}}` 和 `{{user_message}}` |
| Agent 回复路径 | 是 | 合法 JSONPath |
| 挂断信号路径 | 否 | 勾选"无挂断信号"时隐藏 |

| 字段 | 必填 | 说明 |

| 字段 | 必填 | 说明 |
|------|------|------|
| Evaluation 均分阈值 | 是 | 默认 2.0，范围 1.0-3.0，步长 0.1 |

## 十、全局状态与前置校验

### 10.1 发起批测的前置条件

发起批测时系统需校验：

| 条件 | 不满足时的处理 |
|------|--------------|
| 至少存在 1 个 Agent 版本且连接测试通过 | 提示"请先添加并测试 Agent 版本" |
| 至少存在 1 个测试用例 | 提示"请先添加测试用例" |
| 至少配置 1 条 Checklist 或 1 个 Evaluation 维度 | 提示"请先配置裁判标准" |
| 对练模型和裁判模型 System Prompt 不为空 | 提示"请先配置模型 Prompt" |

### 10.2 批测执行流程（系统行为）

```
1. 用户点击「确认发起」
2. 系统为每条用例创建独立会话（生成 UUID 作为 session_id）
3. 对每条用例，循环执行：
   a. 第 1 轮：用例的「首轮发言」作为 user_message 发给 Agent
   b. Agent 返回回复
   c. 检查终止条件：
      - Agent 挂断信号？→ 终止，记录原因"Agent 挂断"
      - 达到最大轮次？→ 终止，记录原因"达到上限"
   d. 将 Agent 回复 + 对话历史发给对练模型，生成下一轮 user_message
   e. 检查对练模型输出中是否包含 [END] 标记：
      - 是 → 终止，记录原因"对练结束"
      - 否 → 回到 步骤 b
4. 对话结束后，将完整对话记录 + Checklist + Evaluation 配置发给裁判模型
5. 解析裁判模型的结构化输出，存储评测结果
6. 所有用例完成后，汇总计算统计数据，更新批测状态为"已完成"
```

### 10.3 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| Agent API 调用失败（网络/超时/格式错误） | 该用例标记为"执行失败"，记录错误信息，不影响其他用例 |
| 对练模型调用失败 | 同上 |
| 裁判模型输出格式不符合预期 | 重试 1 次，仍失败则标记"评判失败"，保留对话记录 |
| 整批测所有用例都失败 | 批测状态标记为"失败" |
| 部分用例执行失败 | 批测状态为"已完成"，通过率仅计算成功执行的用例 |
