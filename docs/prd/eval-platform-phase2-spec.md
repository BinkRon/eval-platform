## 一、二期概述

### 1.1 核心命题

MVP 已验证核心假设：自动化对练+裁判能可靠发现 Agent 的真实问题。但在实际使用中暴露了一个关键瓶颈：

> **配置对练机器人和裁判机器人的门槛过高。** 用户需要将自然的质量直觉拆解为结构化字段（角色卡分段、Checklist 是非题、三级 Rubric），这个过程中信息会变形，配置效率低下。实际操作中，团队更倾向于通过 Claude Code 以 API 写入配置再做微调，而非手动填写表单。

二期的核心命题是：

> **将"AI 生成 + 人工微调"的配置范式产品化**，让用户只需表达评测意图，由构建 Agent 翻译为系统可执行的配置，同时保留人对配置结果的审核与微调能力。

### 1.2 设计理念

两条根本原则驱动本次设计：

**原则一：后端面向 LLM 设计，前端面向人类展示。** 配置的最终消费者是对练模型和裁判模型。数据结构应该让 LLM 最优地理解和执行，前端只是将这些数据以人类可读的方式呈现。

**原则二：LLM 充当人与系统之间的翻译官。** 用户无需理解 prompt 工程或结构化配置规则，只需用自然语言表达评测意图。构建 Agent 负责将意图翻译为系统配置。

### 1.3 验证假设

| 假设 | 验证标准 |
|------|---------|
| H3 效率假设：AI 辅助配置显著降低配置时间 | 同等复杂度的评测方案，配置时间缩短 50% 以上 |
| H4 质量假设：AI 生成的配置质量不低于手动配置 | AI 生成配置驱动的批测结果，裁判判定与人工抽检一致率不低于手动配置 |

### 1.4 关键设计决策

| # | 决策项 | 决策 | 原因 |
|---|--------|------|------|
| 1 | 构建 Agent 入口 | 全局悬浮球，弹出对话界面 | 不侵入现有页面结构，任何页面均可唤起 |
| 2 | 生成粒度 | 分模块顺序生成（对练→裁判） | 两类配置的生成逻辑和输入素材差异大 |
| 3 | 生成确认机制 | 执行前弹出确认卡片，用户确认后写入配置页 | 防止误覆盖，保留用户控制权 |
| 4 | 审核微调方式 | 混合模式：小改直接编辑，大改唤起 Agent | 让用户自行选择成本更低的路径 |
| 5 | 文件处理 | 项目级持久关联 | 避免重复上传，支持增量配置迭代 |
| 6 | 构建 Agent 归属 | 平台内置，Skill 由平台维护 | 确保生成质量可控，用户无需理解 prompt 工程 |
| 7 | 对话持久性 | 项目级持久对话 | 保留上下文，支持增量配置 |
| 8 | P3 配置页定位 | 保持统一配置页，AI 能力由悬浮球承载 | 最小改动，手动配置路径保留为备选 |
| 9 | 数据存储策略 | 存结构化字段，后端拼接 prompt | 单一数据源，无一致性问题 |
| 10 | 对练用例字段简化 | name + sparring_prompt 必填，其余选填或系统默认 | 减少配置负担 |
| 11 | 裁判 Eval 维度 | 从固定三级 Rubric 改为灵活的评判指引文本 | 释放表达力，适配 AI 生成 |
| 12 | 模型 System Prompt | 补充平台默认配置 | 当前缺失，需补全 |

### 1.5 二期变更范围总览

| 模块 | 变更性质 | 影响程度 |
|------|---------|---------|
| 构建 Agent（悬浮球 + 对话 + 确认卡片） | **全新模块** | 主要开发量 |
| P3 配置页 — 用例区块 | 字段简化 | 中 |
| P3 配置页 — 裁判 Eval 区块 | 字段重构 | 中 |
| P3 配置页 — 模型配置区块 | 补充默认 Prompt | 小 |
| P2 工作台 | 新增悬浮球入口 | 小 |
| P5 对话剧场 — 幕后信息 | 适配新 prompt 结构 | 小 |
| 批测执行引擎 — prompt 组装 | 逻辑调整 | 小（简化） |
| 数据模型 | 字段级重构 + 新增实体 | 中 |

**不变的部分：** P1 项目列表页、P2 工作台主体布局、P4 用例概览、P5 对话剧场核心结构、批测执行流程、异常处理逻辑。

## 二、构建 Agent

### 2.1 产品定位

构建 Agent 是平台内置的 AI 助手，充当用户与评测系统之间的翻译官。用户用自然语言表达评测意图，构建 Agent 将其翻译为平台可执行的结构化配置。

构建 Agent **不是**一个通用聊天机器人。它有明确的任务边界：帮助用户配置对练用例和裁判标准。它的 Skill（能力定义）由平台维护，用户不可修改。

### 2.2 入口：悬浮球

**位置：** 页面右下角悬浮球，全局可见（所有项目内页面：P2、P3、P4、P5）。

**状态：**

| 状态 | 表现 |
|------|------|
| 默认 | 悬浮球图标，静态 |
| 有未读消息 | 悬浮球上方出现未读标记（数字），并临时弹出消息气泡预览（3 秒后自动收起） |
| 对话展开 | 悬浮球变为对话面板（右下角弹出，不遮挡主内容区） |

**交互：**
- 点击悬浮球 → 展开对话面板
- 点击面板外区域或收起按钮 → 收起为悬浮球（对话不丢失）
- 对话面板可拖拽调整大小（可选，非必须）

### 2.4 确认卡片组件

确认卡片是构建 Agent 与用户之间的"握手协议"。在 Agent 执行任何写入操作之前，必须通过确认卡片获得用户授权。

**设计原则：** 这是一个可扩展的组件框架。当前用于配置写入确认，未来可扩展到任何需要用户决策的场景。

#### 卡片类型一：生成确认

Agent 完成配置生成后，展示生成结果摘要并请求写入授权。

```
┌──────────────────────────────────────────────┐
│  ✨ 生成完成 — 对练用例（5 条）                │
│                                                │
│  1. 保守型退休客户                              │
│     "喂？" → 58 岁，事业单位退休，风险厌恶…    │
│  2. 激进型年轻投资者                            │
│     "喂？" → 28 岁，互联网从业者，追求高收益…  │
│  3. 对银行高度不信任的客户                      │
│     "喂，你好" → 邻居因 P2P 亏损…              │
│  4. 完全无金融知识的中年客户                    │
│     "喂？" → 45 岁，工厂工人…                  │
│  5. 有投资经验的挑剔客户                        │
│     "你好" → 35 岁，有 3 年基金投资经验…       │
│                                                │
│  ⚠️ 将写入到「测试用例集」，当前已有 0 条用例   │
│                                                │
│  [展开查看完整配置]                              │
│                                                │
│         [取消]  [修改后再写入]  [✅ 确认写入]   │
└──────────────────────────────────────────────┘
```

**卡片字段：**

| 字段 | 说明 |
|------|------|
| 标题 | 生成类型 + 数量（如"对练用例（5 条）"） |
| 摘要列表 | 每条配置的名称和关键信息预览（1-2 行） |
| 影响提示 | 写入目标 + 当前已有数据数量（让用户了解操作影响） |
| [展开查看完整配置] | 展开后显示每条配置的所有字段完整内容 |
| [取消] | 不写入，回到对话继续讨论 |
| [修改后再写入] | 不写入，Agent 进入修改模式，用户可提出调整意见 |
| [✅ 确认写入] | 执行写入，配置出现在 P3 对应区块 |

#### 卡片类型二：覆盖确认

当写入目标区块已有数据时，额外展示覆盖警告。

```

│         [取消]                  [✅ 确认执行]   │
└──────────────────────────────────────────────┘
```

### 2.5 生成工作流

#### 整体流程

```
用户表达评测意图（自然语言 + 可选文件上传）
  │
  ▼
构建 Agent 理解意图，判断信息是否充分
  │
  ├─ 信息不足 → 弹出澄清卡片或追问
  │                │
  │                ▼
  │             用户补充信息
  │                │
  │                ▼
  │             回到意图理解
  │
  ├─ 信息充分 → 开始分模块生成
  │
  ▼
模块一：生成对练用例
  │
  ├─ 生成完成 → 弹出生成确认卡片
  │               │
  │               ├─ 用户确认写入 → 写入 P3 用例区块
  │               ├─ 用户要求修改 → Agent 调整后重新确认
  │               └─ 用户取消 → 回到对话
  │
  ▼
Agent 询问："用例已写入。是否继续生成裁判配置？"
  │
  ├─ 用户确认 → 进入模块二
  ├─ 用户说"我想先调整一下用例" → 暂停，等用户就绪
  └─ 用户说"裁判配置我自己来" → 结束生成流程
  │
  ▼
模块二：生成裁判配置（Checklist + Evaluation）
  │
  ├─ 生成完成 → 弹出生成确认卡片
  │               │
  │               ├─ 已有裁判配置 → 先弹出覆盖确认卡片
  │               └─ 无已有配置 → 直接展示生成确认卡片
  │
  ▼
配置完成
```

#### 模块生成顺序

| 顺序 | 模块 | 生成内容 | 依赖 |
|------|------|---------|------|
| 1 | 对练用例 | name, sparring_prompt, first_message | 用户意图 + 项目文件 |
| 2 | 裁判配置 | ChecklistItem[], EvalDimension[], pass_threshold | 用户意图 + 项目文件 + 已生成的用例（作为裁判标准设计的参考） |

先生成用例再生成裁判的原因：裁判标准应该与用例场景匹配——如果用例侧重投诉场景，裁判就应该包含情绪处理相关的检查项。

#### 用户自定义顺序

如果用户明确指定了生成顺序（如"我只想先配裁判标准"），Agent 应遵从用户意图。默认顺序仅在用户不指定时生效。

### 2.6 项目文件管理

用户上传的文件关联到项目级别，构成该项目的持久知识库。文件是构建 Agent 的工作素材（对话过程数据），不是评测配置的组成部分，因此文件管理入口在对话面板内，而非配置页。

**支持的文件类型：** PDF、Word（.docx）、纯文本（.txt、.md）、Excel（.xlsx、.csv）

**文件在对话中的使用方式：**

- 用户在对话中上传文件时，文件同时存储到项目文件库
- 构建 Agent 每次被唤起时，自动加载项目已有文件作为上下文
- Agent 可以在对话中引用文件内容（如"根据你上传的话术手册第 3 节..."）

**文件管理入口：** 对话面板标题栏的「📁」按钮，点击弹出项目文件列表浮层，展示已上传文件，支持追加上传和删除。

```
┌──────────────────────────────────────────────┐
│  📁 项目文件                          [上传]  │
├──────────────────────────────────────────────┤
│  📎 理财话术手册v3.pdf      3/10     [删除]   │
│  📎 任务流程配置.json       3/08     [删除]   │
│  📎 历史对话示例.xlsx       3/08     [删除]   │
└──────────────────────────────────────────────┘
```

### 2.7 对话持久性

构建 Agent 的对话记录按项目持久保存。

- 每次打开悬浮球，显示该项目的历史对话，自动滚动到底部
- 用户可以翻看之前的对话记录
- 每次新的交互从当前状态出发，不需要用户重复上下文
- 对话记录独立于配置——配置被修改不会影响对话记录，反之亦然

### 3.2 测试用例集 — 字段简化

#### 用例列表

```
┌────┬──────────────────┬──────────────────────────────────┬──────────────┐
│ #  │ 用例名称          │ 角色描述                          │ 操作          │
├────┼──────────────────┼──────────────────────────────────┼──────────────┤
│ 1  │ 保守型退休客户    │ 58岁，事业单位退休，风险厌恶，对… │ [编辑][删除]  │
│ 2  │ 激进型年轻投资者  │ 28岁，互联网从业者，追求高收益，… │ [编辑][删除]  │
│ 3  │ 无金融知识客户    │ 45岁，工厂工人，完全不懂金融产…  │ [编辑][删除]  │
│ 4  │ 对银行高度不信任  │ 邻居因P2P亏损20万，对金融机构…   │ [编辑][删除]  │
└────┴──────────────────┴──────────────────────────────────┴──────────────┘
```

## 四、构建 Agent 的 Skill 定义

### 4.1 Skill 架构

构建 Agent 的能力由一组 Skill 定义，每个 Skill 对应一个配置模块的生成逻辑。Skill 由平台维护，作为构建 Agent System Prompt 的一部分注入。

| Skill | 职责 | 输出 |
|-------|------|------|
| 用例生成 Skill | 理解业务场景，生成多样化的对练用例 | TestCase[] |
| 裁判配置生成 Skill | 提炼质量标准，生成 Checklist 和 Evaluation 维度 | ChecklistItem[] + EvalDimension[] |

### 4.2 用例生成 Skill

**Skill 内容框架：**

```
## 任务
根据用户描述的业务场景和上传的文件，生成一组对练测试用例。

## 输出格式
每条用例包含以下字段：
- name（字符串，必填）：简短的用例名称，应能体现角色的核心特征
- sparring_prompt（字符串，必填，markdown 格式）：完整的角色描述
- first_message（字符串，选填，默认"喂？"）：对练机器人的开场白

## sparring_prompt 的质量标准
一个好的角色描述应包含以下要素（不必严格按此结构，但需覆盖）：
- **角色身份**：年龄、职业、经济状况等基本画像
- **核心诉求**：这次对话要解决什么问题
- **性格与沟通风格**：说话方式、表达习惯、对专业术语的接受度
- **行为规则**：在特定条件下的反应模式（如"Agent 连续推销时会不耐烦"）
- **背景故事**（如有）：影响角色态度的过往经历

## 生成原则
- **多样性**：用例集应覆盖不同类型的用户画像，避免同质化
- **压力测试**：至少包含 1-2 个对 Agent 构成挑战的角色（如情绪化、不信任、知识空白）
- **真实感**：角色的行为逻辑应自洽，不要设定矛盾的特征
- **场景覆盖**：如果用户上传了话术手册或任务流程，用例应覆盖其中的关键分支

## 数量指引
- 用户未指定数量时，默认生成 5-8 条
- 可根据业务复杂度调整
```

### 4.3 裁判配置生成 Skill

**Skill 内容框架：**

```
## 任务
根据用户描述的质量关注点和上传的文件，生成 Checklist 检查项和 Evaluation 评判维度。

## Checklist 输出格式
每条检查项包含：
- content（字符串）：一个可以回答"是"或"否"的明确问题
- level（枚举：required / important）：
  - required（🔴必过）：红线项，违反即不通过
  - important（🟡重要）：影响整体评分但不一票否决

## Checklist 质量标准
- **可判定**：基于对话内容可以明确回答是/否，不依赖外部信息
- **不重叠**：检查项之间不应有显著重复
- **分级合理**：红线项应限于合规底线和严重质量问题，不宜过多（建议不超过总数的 1/3）
- **粒度适中**：不宜过细（导致数量爆炸）或过粗（失去检查价值）

## Evaluation 输出格式
每个维度包含：
- name（字符串）：维度名称，简短明确
- judge_prompt_segment（字符串，markdown 格式）：评判指引

## Evaluation judge_prompt_segment 的质量标准
一个好的评判指引应包含：
- **维度定义**：这个维度评判的是什么
- **评分锚点**：1 分、2 分、3 分各对应什么水平的表现（必须包含，裁判模型依赖此锚点输出评分）
- **边界情况**（建议包含）：容易误判的情况及正确处理方式
- **正反例**（可选）：典型的好表现和差表现描述

## 生成原则
- **Checklist 与 Evaluation 互补**：Checklist 覆盖硬性合规要求，Evaluation 覆盖软性质量维度
- **维度正交**：Evaluation 维度之间应尽量独立，避免高度相关
- **总量可控**：Checklist 建议 3-8 条，Evaluation 维度建议 2-5 个
- **参考用例**：如果已有对练用例，裁判配置应与用例场景匹配

## 通过阈值
- 默认 pass_threshold 为 2.0（3 分满分）
- 如果用户的质量标准明显偏严格或偏宽松，可建议调整
```

### 4.4 Skill 的维护与迭代

Skill 定义存储在平台配置中，不暴露给用户。平台团队基于以下信号迭代 Skill：

- 用户对生成结果的修改频率和修改内容（高频修改说明 Skill 的某些方面不够好）
- 裁判判定与人工抽检的一致率变化
- 用户在构建 Agent 对话中的反馈
