## 一、MVP 概述

### 1.1 核心使命

> 证明自动化对练+裁判能可靠地发现 Agent 的真实问题，并且配置和使用的门槛是可接受的。

### 1.2 验证假设

| 假设 | 验证标准 |
|------|---------|
| H1 价值假设：评测能发现真实问题 | 裁判 Checklist 判定与人工抽检一致率达到可接受水平 |
| H2 体验假设：配置和使用门槛可接受 | 用户能在合理时间内完成首次配置并跑通一轮批测 |

### 1.3 关键设计决策

| 项 | 决策 |
|----|------|
| 项目与 Agent 关系 | 一个项目支持多个 Agent 版本 |
| 用例类型 | 仅角色驱动（L1） |
| 裁判评判时机 | 仅终局评判（对话结束后统一评判） |
| 裁判评判体系 | 双轨：Checklist + Evaluation（简化版） |
| 通过条件 | 固定规则：所有🔴必过 Checklist 通过 + Evaluation 均分 ≥ 阈值 |
| 模型配置层级 | 项目级，对练模型与裁判模型分别配置 |
| 对话终止机制 | 三层：Agent 挂断信号 / 对练机器人自然结束 / 最大轮次兜底 |
| API 协议 | 仅基础协议（纯对话接口） |

## 二、信息架构

### 2.1 设计原则

**原则一：执行为主、配置为辅。**
用户最高频的操作是发起批测和查看结果。配置是低频操作，应"随时可达但不喧宾夺主"。

**原则二：纵深下钻、不平铺。**
批测列表 → 用例概览 → 对话剧场，是层层深入的结构，不是并列关系。

**原则三：一屏掌握全局。**
进入项目后的第一个画面，应能同时看到配置状态和批测结果，而非某个配置表的空列表。

### 2.2 整体结构

```
P1 项目列表页
  └─ P2 项目工作台（= 批测中心，无 Tab 切换）
       ├─ 顶部：配置摘要栏（各项可点击进入配置页）+ [发起批测]
       ├─ 主体：批测记录列表
       │
       ├─ 下钻 → P3 项目配置页（从摘要栏点入）
       │    ├─ Agent 版本管理
       │    ├─ 测试用例集管理
       │    ├─ 裁判配置
       │    └─ 模型配置
       │
       └─ 下钻 → P4 用例概览（从批测列表点入）
            ├─ 统计摘要
            ├─ 用例结果列表
            │
            └─ 下钻 → P5 对话剧场（从用例列表点入）
                 ├─ 左：对话区（实时/回放）
                 └─ 右：评判区（裁判结果 + prompt 透视）
```

### 2.3 页面清单

| # | 页面 | 层级 | 入口 |
|---|------|------|------|
| P1 | 项目列表页 | 顶层 | 平台首页 |
| P2 | 项目工作台 | 二级 | 从 P1 点击项目 |
| P3 | 项目配置页 | 二级（P2 下钻） | 从 P2 配置摘要栏点击 |
| P4 | 用例概览 | 三级（P2 下钻） | 从 P2 批测列表点击 [查看] |
| P5 | 对话剧场 | 四级（P4 下钻） | 从 P4 用例列表点击 [进入剧场] |

### 4.3 配置摘要栏

顶部的配置摘要栏是项目所有配置的"仪表盘"，一眼看清全部就绪状态。

**四个摘要卡片：**

| 卡片 | 展示内容 | 点击行为 |
|------|---------|---------|
| Agent 版本 | 版本数量 + 就绪状态 | 跳转 P3 并定位到 Agent 版本区块 |
| 测试用例 | 用例数量 + 就绪状态 | 跳转 P3 并定位到用例集区块 |
| 裁判配置 | Checklist 数 + 维度数 + 阈值 + 就绪状态 | 跳转 P3 并定位到裁判配置区块 |
| 模型配置 | 对练/裁判模型名称和温度 + 就绪状态 | 跳转 P3 并定位到模型配置区块 |

**就绪状态判定：**

| 配置项 | ✅ 就绪条件 | ⚠️ 未就绪条件 |
|--------|-----------|-------------|
| Agent 版本 | 至少 1 个版本且连接测试通过 | 无版本，或所有版本连接测试未通过 |
| 测试用例 | 至少 1 条用例 | 无用例 |
| 裁判配置 | 至少 1 条 Checklist 或 1 个 Evaluation 维度 | 均为空 |
| 模型配置 | 对练和裁判模型均已选择且 System Prompt 不为空 | 任一项缺失 |

**[发起批测] 按钮状态：**
- 四项全部 ✅ → 按钮可用
- 任一项 ⚠️ → 按钮置灰，hover 提示"请先完成以下配置：{未就绪项列表}"

点击 [发起批测] 后弹出确认面板：

```

│  发起批测                                                      │
├──────────────────────────────────────────────────────────────┤
│                                                                │
│  选择 Agent 版本：[v2.1 - prompt 优化版 ▾]                    │
│  （仅展示连接测试通过的版本）                                   │
│                                                                │
│  实验方案摘要：                                                │
│  · 测试用例：8 条                                              │
│  · Checklist：3 条（2🔴 1🟡）                                 │
│  · Evaluation：3 维度                                          │
│  · 对练模型：Claude Sonnet 4 (temp 0.8)                       │
│  · 裁判模型：Claude Sonnet 4 (temp 0.3)                       │
│                                                                │
│  并发数：[3]                                                   │
│                                                                │
│                                    [取消]  [确认发起]          │
└──────────────────────────────────────────────────────────────┘
```

**确认后行为：**
1. 批测列表新增一行，状态"运行中"
2. **自动跳转到该批测的 P4 用例概览页**——用户无需手动点击 [查看]
3. 用户可实时观测批测进度

### 4.5 批测记录列表

| 字段 | 说明 |
|------|------|
| 状态 | ● 已完成 / ◐ 运行中（附进度百分比）/ ● 失败 |
| Agent 版本 | 本次批测使用的 Agent 版本名称 |
| 用例数 | 本次批测包含的用例数量 |
| 进度 | 已完成用例数 / 总用例数 |
| 通过率 | 通过用例数 / 已完成用例数（运行中时为动态值；失败时显示"—"）|
| 操作 | [查看] → 跳转 P4 用例概览 |

- 按时间倒序排列
- 运行中的批测也可点击 [查看] 进入 P4 实时观测

**连接测试交互：**

2. 展示：请求状态、原始响应 JSON（折叠）、按 JSONPath 解析出的 Agent 回复（高亮）、挂断信号解析结果
3. 用户确认解析正确后保存

**字段规则：**

### 7.6 对话区和评判区的协同关系

左右双栏的核心价值在于**对照阅读**：用户可以一边看对话过程，一边看裁判怎么评的，理解"评分和表现的对应关系"。当裁判总结说"缺少需求诊断环节"时，用户可以立刻在左侧对话中找到——确实，Agent 第一句就直接推荐了产品。

## 八、批测执行流程（系统行为）

### 8.1 正常流程

```
1. 用户在 P2 点击「发起批测」→ 选择 Agent 版本 → 确认
2. 系统创建批测记录，状态"运行中"
3. 自动跳转到 P4 用例概览
4. 对每条用例（按并发数控制）：
   a. 创建会话（生成 UUID 作为 session_id）
   b. 第 1 轮：用例的「首轮发言」作为 user_message 发给被测 Agent
   c. Agent 返回回复
   d. 检查终止条件：
      - Agent 挂断信号匹配？→ 终止，记录"Agent 挂断"
      - 达到最大轮次？→ 终止，记录"达到上限"
   e. 将 Agent 回复 + 对话历史发给对练模型，生成下一轮 user_message
   f. 检查对练输出是否包含 [END] 标记：
      - 是 → 终止，记录"对练结束"（[END] 不展示给用户）
      - 否 → 回到步骤 c
5. 对话结束后，将完整对话 + Checklist + Evaluation 配置发给裁判模型
6. 解析裁判模型的结构化输出，存储评测结果
7. 所有用例完成后，汇总统计，更新批测状态为"已完成"
```

### 8.2 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| Agent API 调用失败（网络/超时/格式错误） | 该用例标记"执行失败"，记录错误信息，不影响其他用例 |
| 对练模型调用失败 | 同上 |
| 裁判模型输出格式不符合预期 | 重试 1 次，仍失败则标记"评判失败"，保留对话记录 |
| 整批测所有用例都失败 | 批测状态标记"失败" |
| 部分用例执行失败 | 批测状态"已完成"，通过率仅计算成功执行的用例 |

## 十、数据实体关系

```
Project（项目）
  │
  ├─ has many → AgentVersion（Agent 版本）
  │                ├─ name, description
  │                ├─ endpoint, method, auth_type, auth_token
  │                ├─ request_template
  │                ├─ response_message_path (JSONPath)
  │                ├─ end_signal_path (JSONPath, nullable)
  │                ├─ end_signal_value (nullable)
  │                └─ connection_status
  │
  ├─ has many → TestCase（测试用例）
  │                ├─ name
  │                ├─ first_message
  │                ├─ persona_background
  │                ├─ persona_behavior (nullable)
  │                └─ max_rounds
  │
  ├─ has one  → JudgeConfig（裁判配置）
  │                ├─ pass_threshold
  │                ├─ has many → ChecklistItem
  │                │                ├─ content
  │                │                ├─ level (required / important)
  │                │                └─ sort_order
  │                └─ has many → EvalDimension
  │                                 ├─ name
  │                                 ├─ description (nullable)
  │                                 └─ level_definitions (JSON: 3级)
  │
  ├─ has one  → ModelConfig（模型配置）
  │                ├─ sparring_provider, sparring_model
  │                ├─ sparring_temperature, sparring_max_tokens
  │                ├─ sparring_system_prompt
  │                ├─ judge_provider, judge_model
  │                ├─ judge_temperature, judge_max_tokens
  │                └─ judge_system_prompt
  │
  └─ has many → BatchTest（批测）
                   ├─ agent_version_id
                   ├─ status (running / completed / failed)
                   ├─ concurrency
                   ├─ created_at
                   ├─ config_snapshot (JSON: 冻结的实验方案快照)
                   │
                   └─ has many → TestResult（用例结果）
                                    ├─ test_case_id
                                    ├─ status (success / failed)
                                    ├─ conversation (JSON: 完整对话)
                                    ├─ actual_rounds
                                    ├─ termination_reason
                                    ├─ checklist_results (JSON)
                                    ├─ eval_scores (JSON)
                                    ├─ judge_summary (text)
                                    ├─ passed (boolean)
                                    ├─ sparring_prompt_snapshot (text)
                                    └─ judge_prompt_snapshot (text)
```

注意两个快照字段：

- `BatchTest.config_snapshot`：冻结发起批测时的完整实验方案（用例集 + 裁判配置 + 模型配置）。后续修改配置不会影响已完成的批测结果的解读。
- `TestResult.sparring_prompt_snapshot` / `judge_prompt_snapshot`：冻结实际发送给对练/裁判模型的完整 prompt。供对话剧场的"幕后信息"展示使用。
