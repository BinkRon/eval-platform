# 项目总体规划

> **[归档文档]** 本文档为跨项目（评测平台 + 编排平台）总规划。评测平台的开发进度请看 `docs/progress.md`。
>
> 文档性质：项目顶层规划，串联各子系统
> 最后更新：2026-03-06

## 一、产品愿景

向投资人证明：**我们的平台能造出有价值的 Agent 产品，并且能系统化地验证 Agent 的质量**。

通过银行产品营销场景，展示三个递进的能力：

1. **可评测**：Agent 的表现可以通过对练 + 裁判机制自动化评测
2. **可编排**：Agent 的对话流程是可视化设计的，非硬编码
3. **可观测**：Agent 运行时的完整决策链路透明可见

## 二、两大子系统

### 2.1 对话 Agent 评测平台

**定位**：独立通用的评测产品，解决"Agent 做得好不好"的问题。

- 不依赖任何特定 Agent 平台的内部实现
- 任何对话机器人只要暴露对话 API 就能接入评测
- 核心能力：对练机器人模拟用户 + 裁判机器人自动评判

**详细设计**（位于独立项目 `eval-platform`）：
- 完整框架：`eval-platform/docs/eval-platform-framework.md`
- MVP 需求：`eval-platform/docs/prd/eval-platform-mvp-spec.md`

### 2.2 任务型 Agent 编排平台

**定位**：可视化编排 + 运行调试平台，解决"Agent 怎么造"的问题。

- 画布式流程编排（节点 + 路由）
- 双模型架构（话术回复 + 反思纠偏）
- 对话/调试双模式运行时可观测

**详细设计**：
- MVP 需求：[orchestration-mvp-spec.md](orchestration-mvp-spec.md)

### 2.3 两者的关系

```
┌─────────────────────┐         ┌─────────────────────┐
│  编排平台             │         │  评测平台             │
│                     │         │                     │
│  画布 + 节点配置      │         │  用例 + 裁判配置      │
│  对话/调试运行        │         │  对练 + 批测          │
│  ↓                  │         │  ↑                  │
│  对话 API ──────────│────────→│  被测 Agent 接入      │
│                     │         │                     │
│  [评测入口] ─────────│────────→│  项目工作台           │
└─────────────────────┘         └─────────────────────┘
```

- **松耦合**：评测平台通过标准对话 API 接入被测 Agent，不依赖编排平台内部实现
- **集成点**：编排平台提供"评测入口"按钮，跳转到评测平台对应项目
- **增强协议**：编排平台可选择在对话 API 响应中附带 `trace` 元数据，评测平台透传给裁判模型以实现灰盒诊断（Phase 3）

## 三、分阶段路线图

### Phase 1：评测平台 MVP（当前阶段）

**目标**：证明自动化对练 + 裁判能可靠发现 Agent 的真实问题。

**核心交付**：
- 项目管理 + Agent 多版本接入
- 角色驱动型用例集
- Checklist + Evaluation 双轨裁判
- 批测执行 + 结果详情

**验证假设**：
| 假设 | 验证标准 |
|------|---------|
| 评测能发现真实问题 | 裁判判定与人工抽检一致率达到可接受水平 |
| 配置和使用门槛可接受 | 用户能在合理时间内完成首次配置并跑通一轮批测 |

**为什么先做评测**：
- 评测平台是独立产品，用户群更广——任何做了对话机器人的团队都有需求
- 编排平台已有基础代码（General-Agent），评测是从零开始，优先补短板
- 评测结果能直接反哺编排平台的 Agent 优化，先有评测才能形成闭环

### Phase 2：编排平台 MVP

**目标**：展示 Agent 的可视化编排和全链路可观测能力。

**核心交付**：
- 画布展示 + 节点高亮 + 跳转切换
- 节点配置面板（prompt、工具绑定、路由）
- 全局配置（话术回复模型、反思模型）
- 对话/调试双模式抽屉 + 详情抽屉
- 反思异步流式展示 + 纠偏注入

**前置条件**：基于现有 General-Agent 代码库改造后端，前端完全重做。

### Phase 3：集成 + 增强

**目标**：打通编排和评测的完整闭环，形成 Demo 叙事线。

**核心交付**：
- 编排平台内嵌评测入口，一键跳转
- 增强协议：编排平台在对话 API 中附带 trace 元数据
- 评测平台支持灰盒诊断（trace 透传给裁判）
- 评测结果反哺编排优化（正反样本 → prompt 改进素材）

**Demo 叙事线**：
> "这是我们搭好的营销 Agent 流程（编排）→ 看它跑起来有多聪明（运行 + 调试）→ 用机器人批量检验它好不好（评测）→ 根据评测结果定向优化（闭环）"

## 四、技术方向概要

| 子系统 | 后端 | 前端 |
|--------|------|------|
| 评测平台 | 新建：对练运行器、裁判运行器、批测调度、项目/用例/结果数据管理 | 新建：React + TypeScript + Vite |
| 编排平台 | 改造：基于 General-Agent 代码库，复用 graph runtime、model adapter、tool 体系 | 新建：React + TypeScript + Vite + React Flow |

两个前端可共享技术栈和基础组件库，后续集成时成本低。
