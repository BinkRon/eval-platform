## 一、产品定位

### 1.1 从任务型 Agent 平台中解耦

评测平台作为独立产品存在，不依赖任务型 Agent 平台的内部实现。核心理由：

- **评测与被测对象天然松耦合**：评测只关心"我给你一句话，你回我一句话，回得好不好"，不关心 Agent 内部的编排、路由、模型架构
- **用户群更广**：任何做了对话机器人的团队——不管底层是规则引擎、RAG、还是 Agent 编排——都有评测需求
- **前端兼容成本低**：任务型 Agent 平台只需保留一个评测入口，跳转到评测平台的独立页面即可

### 1.2 平台愿景

不只是一个"跑分工具"，而是一个**对话质量持续优化的闭环系统**：

> 制造刺激 → 执行实验 → 做出判断 → 输出诊断 → 沉淀资产 → 反哺优化

---

## 二、第一性原理：对话 Agent 评测的本质

评测的本质是：**在受控条件下，观察 Agent 的行为，判断这些行为是否符合预期。**

这件事需要解决三个根本问题，每个问题背后都有一组核心张力：

| # | 根本问题 | 核心张力 | 本质矛盾 |
|---|---------|---------|----------|
| 1 | 怎么制造足够真实的刺激 | 真实性 vs 可控性 | 越像真人越控制不了，越可控越不像真人 |
| 2 | 怎么做出足够可靠的判断 | 多维性 vs 自动化 | 想评判的维度越多越细，越难统一自动完成 |
| 3 | 怎么让结果足够可行动 | 诊断深度 vs 系统耦合 | 越想定位内部原因，越依赖被测系统的信息 |

以下逐一展开分析与应对策略。

---

## 三、张力一 · 刺激的真实性 vs 可控性

### 3.1 问题分析

对于任务型对话 Agent，"刺激"有一个关键特殊性：用户带着意图来，且意图在对话过程中会演变。对练机器人不只是"说话像用户"，它需要**模拟一个有内在状态转移的角色**。

角色的完整描述包含：

- **初始意图**（我想了解理财产品）
- **隐含属性**（风险偏好保守、对银行有不信任感）
- **行为倾向**（容易被打断就会不耐烦、需要被尊重才愿意继续）
- **状态转移规则**（如果 Agent 连续推销两次，就会转为抵触）

### 3.2 应对策略：三层用例体系

不在一种机制里折中，而是分层——让不同层服务不同评测目的。

| 层级 | 名称 | 核心控制要素 | 适用场景 | 可复现性 | 真实感 |
|------|------|------------|---------|---------|--------|
| L1 | 角色驱动 | 角色卡（背景、性格、意图、行为特征） | 探索性测试，发现未知问题 | 低 | 高 |
| L2 | 剧情驱动 | 角色卡 + 关键转折点序列（轮次范围触发） | 定向压力测试，验证特定场景 | 中 | 中 |
| L3 | 脚本驱动 | 完整预设对话序列（或部分锚点+自由段） | 回归测试，变更前后对比 | 高 | 低 |

**设计要点：**

- 三层可在同一测试集中混合使用
- L2 的转折点触发时机用**轮次范围**而非精确轮次，保持鲁棒性
- L3 可支持混合模式：部分轮次固定、部分轮次自由生成
- L3 的主要来源是从 L1/L2 的批测结果中"萃取"而来（见第六章）

## 四、张力二 · 判断的多维性 vs 自动化

### 4.1 问题分析

任务型对话 Agent 的质量是多层次概念：

| 层次 | 关注点 | 示例 | 判定时机 |
|------|--------|------|---------|
| 任务层 | 事情办成了没有 | 客户是否被引导到合适产品 | 对话结束后 |
| 过程层 | 过程合不合规 | 是否跳过了风险评估步骤 | 对话结束后 |
| 话术层 | 每句话说得好不好 | 语气、准确性、合规性 | 每轮实时 |

所有评判维度可归为两大类，处理方式应不同：

| 类型 | 本质 | 示例 | 最优机制 |
|------|------|------|---------|
| Checklist 类 | 事实核查，答案二值 | "是否询问了风险偏好？" | 结构化检查规则 + LLM 提取 |
| Evaluation 类 | 主观评价，需上下文 | "沟通技巧是否得体？" | LLM 裁判 + 评判框架引导 |

### 4.2 应对策略：双轨评判

**Checklist 轨道：**

- 用户定义一组检查项，每项是一个可明确回答的问题
- 每项有**级别**：🔴必过（红线，一票否决）/ 🟡重要（影响评分但不致命）
- 结果是结构化的，天然可聚合统计
- 跑 100 条剧本后能直接看到"80% 的对话都没有做风险评估"

**Evaluation 轨道：**

- 用户定义若干评判维度
- 每个维度有**等级定义 + 典型表现描述**（Rubric），而非一段笼统 prompt
- 平台自动将 Rubric 组装为结构化裁判 prompt
- 改一个维度的标准不需要重写整个 prompt

**通过条件：**

可配置公式，如 `[所有🔴必过 Checklist 全部通过] 且 [Evaluation 均分 ≥ 2.0]`，松紧度可调。

**评判时机：**

| 阶段 | 关注点 | 时机 |
|------|--------|------|
| 逐轮评判 | 单轮话术质量、合规性、幻觉检测 | 每轮 Agent 回复后 |
| 终局评判 | 任务达成度、流程合规性、整体质量 | 对话结束后 |
| 交叉分析 | 跨剧本共性问题、薄弱环节聚类 | 一次批测全部完成后 |

## 五、张力三 · 诊断深度 vs 系统耦合

### 5.1 问题分析

Agent 表现不好，根因可能在非常不同的地方：

| 表现症状 | 可能根因 |
|---------|----------|
| 答非所问 | 节点 prompt 目标描述不清 |
| 跳过必要步骤 | 路由条件设置过于宽松 |
| 编造产品信息 | 工具调用未触发，模型在"幻觉" |
| 忽视负面情绪 | 执行策略缺少情绪应对方案 |
| 回复重复单调 | 温度太低，或 prompt 缺少变化引导 |

评测平台作为独立产品不应强依赖被测系统的内部信息，但有这些信息时诊断能力会大幅提升。

### 5.2 应对策略：双层 API 协议

**基础协议（Basic Protocol）— 任何对话机器人都能对接：**

```
请求：{ session_id, user_message }
响应：{ assistant_message }
```

评测平台在此层做**黑盒诊断**：仅从对话文本推断问题。

**增强协议（Enhanced Protocol）— 可选，提供过程元数据：**

```
请求：{ session_id, user_message }
响应：{
  assistant_message,
  metadata: {
    current_node: "需求诊断",
    tools_called: [{ name: "查询客户画像", result_summary: "..." }],
    route_decision: { from: "开场", to: "需求诊断", reason: "..." },
    reflection: { triggered: true, correction: false }
  }
}
```

评测平台在此层做**灰盒诊断**：将评测结果与 Agent 内部过程关联，定位到具体环节。

**策略要点：**

- 基础协议为必选，增强协议为可选
- 自家任务型 Agent 平台先用增强协议，作为 Demo 亮点
- 增强协议的字段通过**可配置的 JSONPath 映射**实现，不绑定固定数据结构
- 长远看，增强协议有可能演变为对话 Agent 的可观测性标准（类似 OpenTelemetry）

## 六、数据资产与飞轮效应

### 6.1 核心洞察

每次批测沉淀四类数据：

| 数据类型 | 内容 | 价值特征 |
|---------|------|---------|
| 对话记录 | 对练 ↔ Agent 完整多轮对话 | 大量、结构化、带上下文 |
| 裁判标注 | 逐轮评语 + Checklist + Evaluation + 总结 | 等价于自动化生产的标注数据 |
| 过程元数据 | 节点流转、工具调用、路由决策 | Agent 行为的结构化日志 |
| 实验配置快照 | 用例集版本、裁判配置版本、Agent 版本 | 实验的完整可复现条件 |

如果跑完就只是看一眼，平台价值与使用频率线性相关。如果数据能沉淀、关联、复用，平台就具备**复利效应**。

### 6.2 四条复用路径

**路径 A：萃取回归用例（用例飞轮）**

> 角色/剧情驱动探索 → 发现问题 → 修复 Agent → 将暴露问题的对话"固化"为脚本用例 → 后续每次改动自动回归

用例集自动积累，质量防护网越来越密。

**路径 B：正反样本反哺 Agent 优化（优化飞轮）**

- 高分对话 → 作为 Agent prompt 的 few-shot 示例
- 失败对话 + 裁判诊断 + 对应好对话 → "问题-诊断-正例"三件套 → Agent prompt 优化素材
- 进阶：喂给 LLM 自动生成 prompt 改进建议

**路径 C：人工复核校准裁判（裁判飞轮）**

> 裁判自动标注 → 人类抽样校验（同意/不同意）→ 发现裁判偏差 → 优化裁判配置 → 裁判质量提升 → 人类校验比例降低

评测结果的可信度完全取决于裁判的可信度，此飞轮是平台长期可信赖的基础。

**路径 D：跨项目知识复用（生态飞轮）**

- 裁判配置模板：一个团队打磨好的合规性 Checklist 可被同行业其他团队复用
- 角色卡模板："对金融产品不信任的中老年客户"跨银行/保险/证券场景通用
- 远期形态：评测资产库（模板市场）

### 6.3 飞轮全景

```
          ┌──────────── 路径 A：用例飞轮 ─────────────┐
          │                                            │
          ▼                                            │
   ┌─────────────┐     ┌──────────┐     ┌──────────┐ │
   │  实验设计    │────→│  批测执行  │────→│ 结果分析  │─┤
   │（用例+裁判） │     │          │     │          │ │
   └─────────────┘     └──────────┘     └──────────┘ │
          ▲                                    │       │
          │              路径 B：优化飞轮        │       │
          │         ┌─── 正反样本 → Agent 优化 ◄─┤       │
          │         │                           │       │
          │         ▼                           │       │
          │   ┌──────────┐                     │       │
          │   │ 数据资产  │◄────────────────────┘       │
          │   │          │                              │
          │   └──────────┘                              │
          │         │                                   │
          │         ├── 路径 C：人工复核 → 裁判校准 ──────┘
          │         │
          │         └── 路径 D：跨项目模板复用
          │                      │
          └──────────────────────┘
```

## 七、产品结构

### 7.1 整体信息架构

```
项目列表页
  └─ 项目详情（Tab 或侧边导航切换）
       │
       ├─ 被测配置              ← 定义"测谁"
       │    └─ API 接入 + 协议配置（基础/增强）
       │
       ├─ 实验设计              ← 定义"怎么测"
       │    ├─ 用例集管理（角色/剧情/脚本三类混合）
       │    ├─ 裁判配置（Checklist + Evaluation 双轨）
       │    └─ 对练模型默认配置
       │
       ├─ 批测中心              ← "去测"+"看原始结果"
       │    ├─ 发起批测（选择用例范围 + 运行参数）
       │    ├─ 批测列表（状态、通过率、Checklist 通过率）
       │    ├─ 批测详情（双轨结果 + 对话记录 + 元数据追踪）
       │    └─ 批测对比（两次批测结果 diff）
       │
       ├─ 分析看板              ← "看趋势"+"找规律"
       │    ├─ 通过率趋势
       │    ├─ Checklist / Evaluation 维度分析
       │    └─ 高频问题聚类
       │
       └─ 对话资产库            ← "沉淀"+"复用"
            ├─ 对话片段收藏
            ├─ 正样本集（高分对话，可导出为 few-shot 素材）
            ├─ 回归用例集（从批测对话转化的脚本用例）
            └─ 人工复核记录（裁判判定的同意/不同意）
```

### 7.2 页面一：项目列表页

所有评测项目的入口。每个项目卡片直接展示最关键健康度指标：最近批测通过率、趋势方向、用例数、累计批测次数。用户不用点进去就能判断哪个 Agent 需要关注。

### 7.3 页面二：被测配置

核心设计原则——**渐进式复杂度**：

- 最简路径：填 endpoint + 响应中 assistant_message 的 JSONPath → 即可跑起来
- 进阶配置：认证方式、会话管理（session_id 生成规则、结束信号识别）
- 增强协议：可选启用，通过可配置 JSONPath 映射元数据字段
- `[连接测试]` 按钮：发一条测试消息验证连通性和格式解析

### 7.4 页面三：实验设计

#### 用例集管理

- 用例列表支持按类型（角色/剧情/脚本）筛选
- 新建/编辑时根据用例类型展示不同配置表单
- 角色驱动型：角色卡（首轮发言、背景、行为特征）+ 运行参数
- 剧情驱动型：角色卡 + 转折点序列（轮次范围触发 + 行为指令）
- 脚本驱动型：逐轮消息预设（支持混合模式：部分固定、部分自由）
- 每条用例可独立配置对练模型，或继承项目默认

#### 裁判配置

分两步引导用户配置：

**第一步 — Checklist（硬性检查项）：**
- 每条是一个可明确回答的是非题
- 每条有级别：🔴必过 / 🟡重要
- 支持增删排序

**第二步 — Evaluation（评判框架）：**
- 定义若干评判维度
- 每个维度定义等级（如3级：优秀/合格/不合格）+ 典型表现描述
- 平台自动组装为结构化裁判 prompt

**通过条件：** 可配置公式，组合 Checklist 和 Evaluation 结果。

**裁判模型配置：** 模型选择 + 温度（建议低温）+ 最大 Tokens。

### 7.5 页面四：批测中心

#### 发起批测

- 选择用例范围：全部用例 / 手动勾选 / 上次失败用例
- 配置并发数
- 点击发起后进入批测列表

#### 批测列表

- 按时间倒序
- 关键列：时间、用例范围、用例数、状态、通过率、Checklist 通过率
- 操作：查看详情、与另一次批测对比

#### 批测详情

每条用例默认折叠，展开后分区展示：

1. **Checklist 结果区**：逐条检查项的通过/未通过，红线项高亮
2. **Evaluation 评分区**：各维度评分（星级）
3. **对话记录 + 逐轮裁判评语**：对话消息与裁判评语交替展示
4. **裁判总结**：整体评价文本
5. **元数据追踪区**（仅增强协议）：节点流转、工具调用、路由决策的结构化展示，异常项标记⚠️

**新增交互：**
- 每条对话可 `[收藏]`、`[转为回归用例]`
- 裁判判定旁可 `[同意]` / `[不同意]`（人工复核）

#### 批测对比

选择两次批测，做 diff 展示：
- 哪些用例从失败→成功（改进）
- 哪些用例从成功→失败（回退）
- Checklist 各项通过率变化
- Evaluation 各维度均分变化

### 7.6 页面五：分析看板

从宏观到微观的钻取路径：

1. **通过率趋势图**：Agent 整体在变好还是变差
2. **Checklist 通过率分布**：哪条硬性条件最薄弱
3. **Evaluation 维度均分趋势**：哪个维度在进步 / 退步
4. **高频问题聚类**：裁判评语经 LLM 语义聚合后的共性问题模式，附出现频次和关联用例

### 7.7 页面六：对话资产库

- **对话片段收藏**：从批测详情中标记收藏的对话及其裁判标注
- **正样本集**：高分对话，支持导出为 few-shot 素材格式
- **回归用例集**：从批测对话转化而来的脚本用例，自动关联到用例集管理
- **人工复核记录**：汇总所有"同意/不同意"标记，支持按维度统计裁判一致性

## 八、关键设计决策记录

| # | 决策项 | 决策结果 | 原因 |
|---|--------|---------|------|
| 1 | 平台定位 | 独立产品，与任务型 Agent 平台解耦 | 评测天然松耦合，用户群更广 |
| 2 | 被测接入方式 | 标准对话 API，双层协议（基础+增强） | 兼顾通用性和诊断深度 |
| 3 | 刺激控制策略 | 三层用例体系（角色/剧情/脚本） | 分层解决真实性与可控性的张力 |
| 4 | 评判策略 | 双轨评判（Checklist + Evaluation） | 分轨解决多维性与自动化的张力 |
| 5 | 评判框架 | 结构化 Rubric 定义，非自由文本 prompt | 提升裁判稳定性和可迭代性 |
| 6 | Checklist 级别 | 🔴必过 / 🟡重要 | 区分红线和非红线 |
| 7 | 通过条件 | 可配置公式 | 松紧度可调，适配不同阶段 |
| 8 | 诊断深度 | 黑盒（基础协议）/ 灰盒（增强协议） | 不强依赖被测内部信息 |
| 9 | 数据复用 | 四条飞轮（用例/优化/裁判/生态） | 让平台具备复利效应 |
| 10 | 顶层组织单元 | 评测项目（绑定一个被测 Agent） | 支持多 Agent 管理 |
| 11 | 批测执行 | 支持选择性执行（全量/勾选/失败重跑） | 提升迭代效率 |
| 12 | 批测对比 | 两次批测 diff | Agent 迭代的必要反馈机制 |
